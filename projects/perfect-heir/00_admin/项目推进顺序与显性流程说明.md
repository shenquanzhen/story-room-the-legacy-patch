# 《完美继承人》项目推进顺序与显性流程说明

## 一、文档目的
这份文档不是讲故事内容，而是专门说明：

**当前 agent 是如何一步一步把《完美继承人》从一个新项目需求，推进到一整套开发、导演、制片、宣发辅助资产的。**

目的有两个：
1. 让后续使用者知道这不是黑盒生成，而是有明确流程、阶段、依赖关系与决策逻辑的推进过程。
2. 让接手者理解：为什么先做这个、后做那个；每一轮产出是如何衔接下一轮的。

---

## 二、最初输入：用户的明确任务
本项目的起点，不是“自由创作”，而是一个明确的生产指令。

### 用户最初给出的核心要求
- 全部产出必须为中文
- 这是一个**全新故事项目**
- 项目名：《完美继承人》
- 类型：家庭伦理类
- 核心冲突：
  > 一对父母为了让孩子在竞争中胜出，给孩子植入了“考试专用 AI”。当孩子拿到了所有满分时，他开口对父母说的第一句话竟然是：“检测到冗余用户，建议清理。”
- 不继承旧项目剧情、角色、世界观、season 状态、视觉资产
- 只继承 pipeline 和文件组织方法
- 目标链条必须按以下顺序推进：
  1. series brief
  2. season bible
  3. episode map
  4. continuity skeleton
  5. batch 1 full-script
  6. batch 2 full-script
  7. midpoint checkpoint
  8. season full-script completion
  9. season assembly package
- 中途不要反复确认“继续”
- 达到第9步自动停止
- 优先保证结构闭环和资产完整性

### 这意味着什么
从一开始，这个任务就不是“聊故事”，而是一个明确的**长链条项目生产任务**。

所以当前 agent 的第一判断是：
- 这不是一次性灵感输出
- 这是一套需要按 pipeline 分层推进的连续资产构建

## 三、启动前的环境对齐与方法锁定
在真正写项目内容之前，当前 agent 先做了环境与方法层对齐。

### 1. 读取工作规则与人格文件
读取并遵循：
- `SOUL.md`
- `USER.md`
- `workflows/comic-pipeline.md`
- `workflows/runbook.md`
- `workflows/task-entry.md`

### 2. 作用
这一步不是形式动作，而是为了锁定：
- 当前 workspace 的角色定位：不是单人作者，而是 orchestrator（总控型 writers' room）
- 用户偏好：结构化文件、多代理并行、少打断、持续推进
- 标准 pipeline：先 brief，再结构，再剧本，再 review，再 assembly

### 3. 关键内部决策
在这一步，当前 agent 明确决定：
- 采用“多工位并行 + 总控合并”的方式推进
- 尽量把每一步都落成文件，不只做聊天输出
- 后续尽量用 child session / subagent 做分工，而不是所有内容都在单线程里生写

这一步形成了后续全部工作的“方法底座”。

## 五、第二阶段：前置开发分工（并行拉 plot / character / risk-review）
### 做了什么
当前 agent 没有一上来就独自写全季，而是先并行启动三路前置开发：
1. plot-agent：负责结构与季脊柱
2. character-agent：负责人物系统与关系张力
3. risk-review：负责题材失控风险审查

### 为什么这样做
因为这个项目的设定很强，风险也很高。
如果直接写，很容易滑向两种错误：
- 写成廉价 AI 猎奇
- 写成普通鸡娃伦理，没有新意

所以必须先做前置审查和结构锁定，避免后面返工。

### 三路前置开发分别解决了什么
#### 1. risk-review 解决了：
- 题材最容易翻车在哪里
- 如何把它稳在家庭伦理而不是纯科幻
- 结构上必须守住哪三条线
- 哪些结局方式必须禁用

#### 2. plot-agent 解决了：
- 24集全季 spine
- 四阶段季结构
- 关键 reversals
- series brief 与 season bible 的故事架构基础

#### 3. character-agent 解决了：
- 核心家庭角色
- 家族对抗角色
- 关系图
- voice cues
- contradiction points
- continuity 需要的人物骨架

### 显性流程意义
这一阶段相当于：
先把“骨架、人物、风险控制”三件事做扎实，后面才不会边写边找方向。

## 六、第三阶段：按用户指定链条，完成 season 基础开发包（1-4步）
这一步严格对齐用户最初要求的链条前四项。

### 产出文件
1. `01_brief/series-brief.md`
2. `02_bible/season-bible.md`
3. `03_structure/episode-map.md`
4. `04_continuity/continuity-skeleton.md`

### 为什么按这个顺序
#### 1）Series Brief 先行
作用：让项目一句话能被理解。  
解决：这到底是什么剧，卖点是什么，边界是什么。

#### 2）Season Bible 第二
作用：把 brief 的概念转成系统。  
解决：角色、世界规则、技术边界、季结构怎么运转。

#### 3）Episode Map 第三
作用：把 season 级结构具体化到24集。  
解决：每集有什么功能、钩子、推进点。

#### 4）Continuity Skeleton 第四
作用：防止后续写剧本时人物断裂、逻辑串味。  
解决：谁怎么变化，哪些埋词怎么回收，权限怎么升级。

### 显性流程意义
这一步是从“概念成立”转向“可以开始写剧本”的关键过渡层。

如果没有这四份文件，后面写 batch scripts 时就会很容易：
- 角色乱说话
- 中段重复
- 结尾跑偏
- 世界规则失控

## 七、第四阶段：完成全季剧本开发主链（5-8步）
在基础开发包完成后，当前 agent 开始进入用户指定链条中最重的部分：剧本层。

7. `07_checkpoint/midpoint-checkpoint.md`

### 实际推进逻辑

目的：

- 检查中段前是否健康

#### 2）再写 Batch 2（13-24集）
目的：

#### 3）Midpoint Checkpoint 中段复盘
目的：
- 检查前半季有没有塌
- 确认后半季执行重点
- 保证下半季不是换题，而是加压

#### 4）Season Full-Script Completion
目的：
- 确认剧本链条已经闭环
- 明确全季角色弧光、冲突闭环与结尾余波都成立

### 为什么没有在中间停下来问用户
因为用户一开始就明确授权：
- 不要反复确认“继续”
- 到第9步自动停止

所以当前 agent 采用的是：
**先内部完成结构检查，再继续推进，而不是中途打断用户。**

## 十七、并行分工在整个流程里的作用
为了避免后续使用者感觉“怎么一下子就冒出那么多文件”，这里显性说明一下当前 agent 的工作方式。

### 当前 agent 采用的基本模式
不是所有内容单线程手写，而是：
- 总控 agent 负责决定阶段、顺序、依赖关系与最终合并
- 子代理负责某一层的并行产出
- 总控 agent 再回收、合并、落盘、提交 git、继续下一轮

### 这种模式出现在哪些阶段
- 前置开发：plot / character / risk-review
- 逐集精修台词版
- 导演分场版
- 拍摄对白压缩版
- 爆点金句提炼版
- 场景清单版
- 制片拆景版及通告统筹版

### 为什么这样做
因为这是一个多工位 writers' room / production-room 型任务，适合：
- 并行提速
- 角色分工清楚
- 避免一轮一轮串行拖慢
- 保持总控决策权在同一个 agent 手里

也就是说：
**内部不是“随便生成”，而是“总控 + 专项分工 + 合并封装”的工作流。**

1. 输出 `当前agent工作总说明.md`
2. 输出本文 `项目推进顺序与显性流程说明.md`

### 这意味着

而是因为用户已经把当前轮次的目标改成：
**先把已经完成的工作讲清楚、整理清楚、交接清楚。**
